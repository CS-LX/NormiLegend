# 资产索引方案（Asset Index）

> 目标：让 AI 依据提示词查找资产时**更快、更准、几乎不消耗 token**，同时不增加美术的维护负担。
>
> 状态：方案已定稿，开始实施 MVP。

---

## 1. 背景与目标

项目 `assets/` 已有 264 张图片（521MB），命名混杂（哈希名 / 时间戳名 / 中文语义名），
`.meta` 仅含 uuid 无语义信息，且没有任何索引文件。

AI 每次查找资产都要 `ls`/`find` 全量扫描，再逐张 `Read` 图片确认内容，token 消耗极高。
本方案构建一个**语义化资产索引**，通过 `Grep` 关键词检索（只返回命中行），把查找成本降低 1~2 个数量级。

---

## 2. 可行性结论

**高度可行，且是当前性价比最高的优化之一。**

核心机制：**索引文件 + Grep 检索**。Grep 只返回匹配行，查找成本与索引大小无关，可无限扩展。

---

## 3. 现状与痛点

| 项目 | 现状 |
|------|------|
| 资产规模 | `assets/` 521MB，图片 264 张 + 329 个 `.meta` |
| 目录组织 | `assets/image/` 顶层堆 200+ 散文件 + 8 个中文子目录 |
| 命名 | 哈希名(`cgt-...png`) / 时间戳名(`edited_xxx_2026...png`) / 中文名(`白色尖顶教堂横版背景.png`) 三种混杂 |
| `.meta` | 仅 `{"uuid":"..."}`，无分类 / 描述 / 尺寸 |
| 现有索引 | 无 |
| 编辑器分类机制 | `TitleMenu.lua` 的 `ImportTexture(path,name,cat)` 硬编码手动注册，`mapAssets` 硬编码中文数组 |

**AI 当前查找代价**：
- 一次 `ls assets/image/` ≈ 600 行 ≈ 4000~5000 token
- 哈希名看不懂时逐张 `Read` ≈ 每张 1000~1700 token
- 一个会话找 3~5 张 ≈ 10000~15000 token，且仍可能选错

---

## 4. 方案设计

### 4.1 索引格式（单文件，每行一条，Grep 友好）

```
[分类:子标记] 路径 | 名称 | 语义描述 | 尺寸/帧数
```

示例：
```
[bg]        image/edited_城堡星空高清_20260604085456.png | 城堡星空 | 紫色星空哥特城堡远景 | 1920x1080
[seq:12f]   image/char3_run_12f.png | 角色3奔跑 | 12帧横向序列帧
[dlg_whole] image/对话框/整体/诺米-常态.png | 诺米常态 | 角色诺米中性表情整体对话框 | 1600x1075
[dlg_bg]    image/对话框/底图/诺米对话框-宽.png | 宽底图 | 对话框背景宽版
```

### 4.2 检索方式（关键）

**不全文读索引，而用 `Grep` 按关键词命中**——只返回匹配行。
- 找对话框立绘 → `Grep "dlg|对话框|立绘"`，约 100~300 token
- 找冰法少女 → `Grep "冰法"`，一次命中所有相关
- 相比现状 **30~50 倍 token 节省**，描述明确不易选错

### 4.3 存放位置

| 用途 | 文件 | 读取方 | 阶段 |
|------|------|--------|------|
| 服务 AI 查找 | 项目根 `ASSET_INDEX.md` | AI 用 Grep 检索 | P0 |
| 编辑器复用 | `assets/asset_index.json` | 运行时 `File` API | P1 |

P1 的架构红利：编辑器改读索引文件后，可消除 `TitleMenu.lua` 里硬编码的 `ImportTexture`
列表和 `mapAssets` 数组，AI 与编辑器共用一份数据源。

## 5. 生成与维护

### 5.1 字段来源与自动化程度

| 字段 | 来源 | 自动化 |
|------|------|--------|
| path | 扫描 `assets/image/**` | 全自动 |
| 分类 cat | 目录前缀(`平台-`→tile,`背景-`→bg) + 现有 `ImportTexture` 注册 | 全自动 |
| 尺寸 | 读 PNG/JPG 文件头(不解码) | 全自动 |
| 帧数 | 文件名 `_12f_`/`_4f_` 正则 | 全自动 |
| 名称 | 中文文件名直接用；哈希名留空待补 | 半自动 |
| **语义描述** | 中文名可推断；哈希名需 AI 看图补充（一次性） | 半自动 |

### 5.2 生成器

- 一个 headless 脚本（Python，沙箱内由 AI 运行），扫描 + 解析，**离线运行不占会话 token**
- 增量更新：对比已有索引，只为新增/缺失项补条目
- 描述补充：哈希名图片由 AI 批量看图生成（一次性成本）

---

## 6. 维护负担说明（美术视角）

**结论：不需要美术写或懂任何脚本，维护成本压在 AI 一侧。**

- 脚本的写 / 跑 / 修 / 装环境全部在沙箱内由 AI 完成，美术本机什么都不装、不跑
- **不采用 CI / 定时任务**（那类需要人配置和排错，对美术不友好）

| 模式 | 美术要做什么 | 谁排错 | 采用 |
|------|-------------|--------|------|
| CI / 定时脚本 | 配置、看报错、修环境 | 美术 | ❌ |
| **AI 按需触发** | 导入后说一句"更新索引" | AI | ✅ 现在 |
| 编辑器导入即更新 | 什么都不用做 | AI（内置） | 🔜 P1 后 |

诚实说明：哈希名图片(`cgt-xxx.png`)的内容描述无法纯机器自动（需看图），但这件事也由 AI 做；
美术大部分新素材是中文命名，这类完全自动。
